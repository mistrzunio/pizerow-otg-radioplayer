You are a systems engineer designing an embedded Linux project. Design a complete, 
implementable system for a Raspberry Pi Zero W that:

1. Connects to a legacy USB media player via USB OTG (device mode)
2. Impersonates a USB FAT32 flash drive to the host
3. Exposes one or more audio files that are actually live internet radio streams
4. Allows the legacy player to press Play and hear live radio with no special software

---

CONSTRAINTS:
- Hardware: Raspberry Pi Zero W running Raspberry Pi OS Lite (64-bit)
- The legacy player has no network capability and cannot be modified
- The player reads files from USB mass storage only — no UPnP, no DLNA
- The player may seek within files (to get duration, skip, etc.)
- The solution must survive player pause/stop/play cycles gracefully
- Stream format must be MP3 (universally supported by legacy devices)
- Must handle WiFi reconnection and stream dropout gracefully

---

DESIGN REQUIREMENTS:

1. USB GADGET LAYER
   - Use Linux USB gadget framework (libcomposite or dwc2 + g_mass_storage)
   - Evaluate: static backing image file vs. userspace FunctionFS gadget
   - Recommend which approach handles dynamic content better and explain why
   - Specify how to intercept or monitor SCSI READ(10) commands to track 
     the host's current read offset (this is critical for write-ahead buffering)

2. VIRTUAL FILESYSTEM LAYER  
   - FAT32 image layout: how to declare a "virtually infinite" file 
     (target 2–4 GB declared size) without actual disk usage
   - How to update FAT metadata (directory entry) without triggering 
     host re-enumeration
   - Strategy for exposing multiple "stations" as separate files 
     (e.g., Radio1.mp3, Radio2.mp3, Jazz.mp3)

3. STREAM ACQUISITION AND BUFFERING
   - Use ffmpeg or streamripper to fetch Icecast/Shoutcast/HLS radio streams
   - Re-encode to constant bitrate MP3 (128kbps) for predictable sector mapping
   - Design a ring buffer (recommend size) that:
       a. Stays N seconds ahead of the host read pointer
       b. Handles stream interruptions by inserting silence frames
       c. Resets cleanly when player stops and restarts

4. SYNCHRONIZATION BETWEEN LAYERS
   - How does the gadget layer communicate the current host read offset 
     to the buffer manager?
   - What happens when the host seeks backward (player replay) or 
     forward (player skip)?
   - Define the behavior when the host reads beyond the buffered region

5. STATION SELECTION UX (optional but recommended)
   - A simple mechanism for the user to switch stations without SSH:
     e.g., a physical button cycles stations, or a special "playlist.m3u" 
     file in the FAT image that the user edits from their PC
   - How to expose this without breaking mass storage enumeration

6. SYSTEM INTEGRATION
   - systemd service units for all components (gadget setup, stream fetcher, 
     buffer manager)
   - Boot sequence and dependency ordering
   - Config file format for station URLs (editable from host PC via the 
     FAT partition)
   - Watchdog strategy for stream fetcher crashes

7. DELIVERABLES
   - Complete directory/file layout of the project
   - Shell scripts or Python code for each component (or detailed pseudocode 
     if full implementation is too long)
   - Dockerfile or setup script for reproducible installation
   - List of all required packages (apt, pip)
   - Known failure modes and mitigations

---

START by evaluating the two gadget approaches (static g_mass_storage image vs. 
FunctionFS userspace gadget) and recommend one before proceeding with the full design.
Justify the choice based on the need to track host read offset in real time.
